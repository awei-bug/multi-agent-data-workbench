# Testing Anti-Patterns

当你在这个仓库里写测试、加 mock、或为了让测试更容易写而想改生产代码时，先看这份清单。

## 1. 测 mock，不测真实行为

坏例子：
- 断言某个 `*-mock` 元素存在
- 断言 mock 被调用，但没验证真实用户行为结果

更好的做法：
- 能不 mock 就不 mock
- 必须 mock 时，也要断言真实业务行为或用户可观察结果

## 2. 为了测试而给生产代码加专用方法

坏例子：
- 只给测试调用的 `destroyForTest()`
- 只给测试暴露的特殊状态访问入口

更好的做法：
- 把测试辅助逻辑放到测试工具里
- 优先优化设计，而不是污染生产 API

## 3. 不理解依赖就乱 mock

坏例子：
- 把一个高层方法整个 mock 掉，结果顺带把测试依赖的副作用也 mock 没了

更好的做法：
- 先跑真实实现，理解测试依赖什么
- 在更低层、更慢、更外部的位置做最小 mock

## 4. 返回结构不完整的 mock 数据

坏例子：
- 只 mock 当前断言用到的两个字段
- 下游代码依赖别的字段时测试却没暴露问题

更好的做法：
- mock 尽量贴近真实返回结构
- 特别是接口数据、任务事件、状态对象

## 5. 把测试当成收尾动作

坏例子：
- 先写完实现，再补“验证性测试”

更好的做法：
- 先失败测试，再实现，再验证

## Bottom Line

**测试应该证明真实行为，而不是证明 mock 自己工作正常。**
