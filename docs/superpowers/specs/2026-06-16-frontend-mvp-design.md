# 多 Agent 数据分析工作台前端 MVP 设计

**目标**

基于 React 实现一个可交互的前端 MVP 原型，完整覆盖“数据源接入 -> 需求输入 -> Agent 任务流预览 -> 运行监控 -> 人工修正 -> 可视化看板 -> 报告编辑 -> Word/PDF 导出交互”的工作台主流程。在当前目录没有后端代码的前提下，使用前端本地 mock 状态机模拟任务执行、产物生成和导出反馈，为后续接入真实后端预留清晰边界。

## 1. 范围

### 1.1 本期包含

- React 单页应用工作台
- 左侧导航、顶部任务栏、主流程工作区、右侧上下文面板
- CSV/Excel 文件上传交互
- 数据源概览、字段信息、样例数据展示
- 自然语言需求输入与任务预解析结果展示
- 可编辑的 Agent 流程图预览
- 任务运行状态流转与日志展示
- 清洗后数据在线编辑与局部重跑模拟
- 图表卡片展示与图表配置修改
- 结构化报告编辑
- Markdown、Word、PDF 导出交互与下载反馈
- 历史任务列表的基础查看与复用入口

### 1.2 本期不包含

- 真实登录鉴权
- 真实后端接口
- 真实 WebSocket 长连接
- 真实数据库连接
- 真正的多用户隔离
- 真正的 Word/PDF 排版引擎
- 真实对象存储、消息队列、任务调度服务

## 2. 实现策略

采用“纯前端 MVP，内置 Mock 数据与状态机”方案。

理由：

- 当前项目没有现成代码仓库与后端服务，纯前端路径最短。
- 用户要求完整工作台主流程，纯静态页面无法体现任务依赖、运行状态和人工介入。
- 用本地状态机模拟 Agent 执行过程，可以在不引入服务端复杂度的前提下还原核心产品体验。
- 后续接后端时，只需要替换数据层和事件层，不需要推翻 UI 结构。

## 3. 信息架构

应用采用单页工作台形态，减少路由跳转导致的上下文丢失。

### 3.1 顶层布局

- 左侧导航
  - 工作台首页
  - 新建分析任务
  - 历史任务
  - 数据源管理
  - 模板库
- 顶部任务栏
  - 当前任务名称
  - 保存草稿
  - 重新运行
  - 导出
  - 全局状态提示
- 中央主工作区
  - 面向主流程的步骤面板
- 右侧上下文面板
  - 当前 Agent 摘要
  - 日志流
  - 输入输出结果
  - 风险提示

### 3.2 页面组织

虽然是 SPA，但内部仍保留两个主视图：

- 工作台主流程视图
- 历史任务视图

数据源管理和模板库保留为轻量占位页或抽屉入口，不单独展开复杂交互。

## 4. 主流程设计

### 4.1 步骤 1：数据源接入

目标：让用户完成文件上传并看到初始数据概览。

交互：

- 拖拽上传或点击选择 `CSV` / `Excel`
- 显示上传状态、文件名、大小、时间
- 上传成功后展示：
  - 字段列表
  - 字段类型
  - 缺失率
  - 样例数据表格
- 提供“数据库直连”表单入口，但仅做 mock 成功

产物：

- `dataSource`
- `rawSchema`
- `previewRows`

### 4.2 步骤 2：需求输入

目标：将自然语言需求转换为结构化分析目标。

交互：

- 多行输入框
- 文档示例需求快捷填充
- 点击“预解析”后显示：
  - 分析目标
  - 核心指标
  - 建议维度
  - 推荐图表类型

产物：

- `analysisRequest`
- `parsedIntent`

### 4.3 步骤 3：Agent 任务流预览

目标：可视化展示多 Agent 主链路，并允许用户做轻量调整。

默认节点：

- 数据接入 Agent
- 数据清洗 Agent
- 指标计算 Agent
- 可视化 Agent
- 分析推理 Agent
- 报告撰写 Agent
- 审核校验 Agent

交互：

- 流程图节点按依赖顺序排列
- 支持节点启用/禁用
- 支持插入“人工校正”节点
- 支持有限拖拽调整顺序
- 如果调整造成非法依赖，给出前端校验提示

产物：

- `workflowNodes`
- `workflowEdges`

### 4.4 步骤 4：运行监控

目标：让用户看到任务如何逐步执行并产出中间结果。

交互：

- 点击“启动任务”
- 状态机驱动节点流转：
  - `idle`
  - `pending`
  - `running`
  - `success`
  - `warning`
  - `failed`
- 中央显示流程状态
- 右侧显示当前 Agent：
  - 执行日志
  - 耗时
  - 输入摘要
  - 输出摘要
- 支持暂停、重试、终止的 UI 行为模拟

产物：

- `executionState`
- `agentLogs`
- `artifacts`

### 4.5 步骤 5：人工修正

目标：允许用户编辑中间数据并触发局部重跑。

交互：

- 展示清洗后数据表
- 支持修改单元格
- 支持标记异常值
- 支持新增自定义指标说明
- 点击“应用修改”后：
  - 标记受影响节点
  - 仅重跑下游节点
  - 保留上游成功结果

产物：

- `cleanedRows`
- `manualEdits`
- `rerunTargets`

### 4.6 步骤 6：可视化看板

目标：让用户调整图表表达方式。

交互：

- 图表卡片网格
- 每张图支持：
  - 切换图表类型
  - 修改标题
  - 修改配色主题
  - 隐藏或显示
- 图表修改即时同步到报告内容引用

产物：

- `chartConfigs`
- `dashboardLayout`

### 4.7 步骤 7：报告编辑

目标：在系统生成报告草稿的基础上进行可控编辑。

交互：

- 结构化分节展示：
  - 执行摘要
  - 数据概况
  - 核心发现
  - 图表分析
  - 风险与建议
- 支持编辑文案
- 支持插入图表块
- 支持插入数据表块
- 支持新增自定义段落

产物：

- `reportDocument`

### 4.8 步骤 8：导出交互

目标：完整闭环到可下载产物。

交互：

- 提供 `Markdown`、`Word`、`PDF` 三个按钮
- 点击后显示导出进度
- `Markdown` 可直接导出文本文件
- `Word` / `PDF` 使用 mock 异步任务模拟生成
- 导出完成后提供下载链接和最近导出记录

产物：

- `exportJobs`
- `downloadableFiles`

## 7. 技术约束与建议

### 7.1 技术栈

- React
- TypeScript
- Vite
- 状态管理优先选择轻量方案，例如 `Zustand`
- 路由使用 `React Router`
- 图表使用 `ECharts`
- 表格优先使用成熟表格组件或轻量封装
- 样式使用模块化 CSS 或原子化方案，但要统一变量体系

### 7.2 视觉方向

- 保持“分析工作台”专业感，不做营销页风格
- 信息密度偏高，但层级要清晰
- 流程状态、告警和人工介入点必须一眼可见
- 图表区、日志区、编辑区三类信息要明显区隔

### 7.3 模拟机制

- 不用真实 WebSocket
- 用前端定时器和事件分发模拟日志流
- 用本地 JSON 和工厂函数生成不同阶段产物
- 用任务依赖映射模拟局部重跑

## 8. 数据流

### 8.1 正常主链路

1. 用户上传文件
2. 前端解析基础信息并生成预览
3. 用户输入需求
4. 系统生成结构化任务意图
5. 系统生成 Agent 流程
6. 用户启动任务
7. 模拟引擎按顺序推进各 Agent
8. 中间产物写入全局状态
9. 用户修改数据或图表
10. 系统局部重跑依赖节点
11. 系统生成报告草稿
12. 用户编辑并导出

### 8.2 人工介入链路

1. 任务执行到中间节点
2. 用户在数据表或图表区修改内容
3. 系统记录修改来源
4. 系统识别受影响下游节点
5. 系统重新运行必要节点
6. 报告与导出结果同步更新

## 9. 验收标准

### 9.1 交互验收

- 用户能完整走通上传到导出的主链路
- 每个主链路步骤均有明确状态反馈
- Agent 运行时能看到节点流转和日志变化
- 人工修正后能看到局部重跑效果
- 图表修改会同步影响报告展示
- Word/PDF 导出交互可触发、可等待、可下载

### 9.2 工程验收

- 项目可本地启动
- 首屏无报错
- 主流程状态切换稳定
- mock 数据结构清晰，后续可替换 API
- 组件边界明确，避免把所有逻辑堆在单文件

## 10. 风险与处理

### 10.1 范围失控

风险：工作台模块很多，容易从 MVP 演变成完整产品。

处理：只打通一条主链路，其他模块保留轻入口，不展开实现。

### 10.2 交互太假

风险：如果只是静态切页，无法体现多 Agent 协同价值。

处理：必须实现任务状态机、日志流和局部重跑机制。

### 10.3 组件耦合

风险：原型项目常把流程、数据、图表、报告逻辑混在一起，后续难替换真实接口。

处理：从开始就分离 UI 组件、全局状态和 mock 服务层。

## 11. 实施边界结论

这次前端实现的核心不是“做很多页面”，而是“做一个能演示多 Agent 数据分析工作台主流程的交互原型”。任何不直接增强主链路体验的内容，都不进入本期实现。
